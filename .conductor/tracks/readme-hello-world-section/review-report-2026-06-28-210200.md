## 3. Blocking Issue: Path Resolution

I am presenting this to the user rather than editing it unilaterally because it is an intent decision, not a technical fix. Two clean options:

### Option A — Edit the user-config copy
- **Path:** `C:\Users\DaveWitkin\.config\opencode\skill\conductor-pipeline\README.md`
- **Pros:** The file already exists and is well-formed (it is the canonical conductor-pipeline README per its own "Files created by this track" section). No directory creation needed. `Add-Content` appends cleanly. The git diff will only show the workspace, not the user config, so the user-config change is invisible to the repo's `git status` — but that is also a **con** (the change is uncommitted / unsynced with any repo).
- **Cons:** Contradicts the spec's explicit "do not modify" rule. The user config is the runtime config for OpenCode; modifying it means the user has to restart OpenCode to pick up changes. Also, this file is **outside** the workspace repo, so the conductor track's own `git diff --name-only` gate would show zero changes — meaning the plan's "diff is limited to target file" verification would technically pass vacuously, which weakens the verification.
- **Plan changes required:** swap target path everywhere; remove the "do not modify user config" rule; note that `git status --porcelain` will not see the change (consider verifying the file content directly with `Get-Content` / `Select-String` rather than git-based scope checks).

### Option B — Create a workspace-local copy
- **Path:** `C:\development\opencode\skill\conductor-pipeline\README.md`
- **Pros:** Honors the spec's "do not modify user config" rule. The change is fully visible to git. The plan's git-scope verification works as designed (now that I edited it to use `git status --porcelain`).
- **Cons:** Creates a new `skill/conductor-pipeline/` directory and README in the workspace, which becomes a workspace-local shadow of the user-config skill. Requires `New-Item -ItemType Directory -Force` for the parent path before `Add-Content`. The skill-shadowing concept is not discussed anywhere in the spec; the user may not want a duplicate copy of a skill in the repo. If both copies exist, OpenCode's skill discovery rules determine which one is loaded.
- **Plan changes required:** add a Phase 0.5 that creates the parent directory if missing; update Phase 0.2's error recovery to fall through to directory creation (after explicit user confirmation that Option B is the chosen path); keep my Phase 2.3/2.4 edits as-is.

### Recommendation

**Option B** is more consistent with the spec's stated constraints and with the spirit of a "workspace repo" track. **Option A** is operationally simpler but contradicts the spec.

**If neither A nor B is the intended target, the plan must be re-scoped before Stage 4.**
