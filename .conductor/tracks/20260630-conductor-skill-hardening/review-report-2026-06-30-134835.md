## Anti-laziness mandate — dry-run results

I dry-ran EVERY reviewer-added or reviewer-modified PowerShell verification command against (a) the real target where safe, and (b) a temp script that simulates the body the plan will produce. I treated any reviewer-added PowerShell as UNTRUSTED until executed exactly as written.

**Key dry-run findings (BLOCKING bugs found in the original plan):**

1. **`-notlike "*[regex]::Escape()*"` silently fails.** PowerShell `-like` treats `[` and `]` as character-class wildcards. When I ran `-notlike "*[regex]::Escape()*"` against a body that contains the literal text `[regex]::Escape()`, the result was `True` (i.e., `-notlike` reports the body does NOT contain the literal). This means Task 1.1's check would have FAILED the verification step even when the executor correctly inserted the body. Confirmed via direct test (`test20.ps1`).
2. **Task 2.1's 4th pattern is a PARSER ERROR.** The double-quoted PowerShell string `` "`completed_tasks` - map this value to completed executable tasks out of `task_count`" `` ends with a backtick before the closing `"`. In PowerShell, `` ` `` before `"` escapes the quote, leaving the string unterminated. The whole script aborts at parse time (`ParserError: The string is missing the terminator: "`). Confirmed via direct test (`test13.ps1`).
3. **Task 2.1's other backtick patterns silently miss.** Even the 3 patterns that DO parse (`` "`task_count` - count executable implementation task checkboxes only" ``, etc.) have `` `t `` interpreted as a tab character and the trailing lone `` ` `` dropped, so the resulting pattern is `\task_count\ - count ...` (tab + "ask_count" + " - count ..."), which is not in the body. Confirmed via direct test (`test_fix2.ps1`).
4. **Task 3.1's bracket patterns silently fail.** Same as (1): `` -notlike "*[string]::Replace()*" `` and `` -notlike "*[regex]::Escape*" `` fail to match the body. All eight patterns in Task 3.1's check were tested; every one returns `True` for `-notlike` even when the body contains the literal text (`test20.ps1`).
5. **Task 3.1's dollar/backtick pattern is also broken.** `` "`$text.Replace(`$old,`$new)" `` becomes `$text.Replace($old,$new)` after PowerShell parses the backticks (which escape the dollar signs to literal). The body's literal backticks are not in the parsed pattern, so `-notlike` misses. Confirmed via direct test (`test19.ps1`).

**Fix approach applied:** All reviewer-modified body-content checks now use **single-quoted PowerShell string patterns** (which preserve literal backticks) and **`[string]::Contains()` literal substring matching** (which is immune to `[`/`]` wildcards). I also added a top-level "Body-writing rule" constraint that tells the executor to write bodies via `@''...''@` (single-quoted here-string) or `[System.IO.File]::WriteAllText` with a literal `$literal` string — never a double-quoted here-string holding a body that contains backticks, because `` `t `` becomes a tab and `` ` `` before `"` escapes the closing quote.

All rewritten checks were dry-run against simulated post-insertion bodies and reported `... body valid` correctly. See "Dry-run results per task" below.
