# Semantic Search and Community Detection

This document defines how to use search tools and heuristics to discover related but
unlinked nodes, identify entity clusters, and apply LLM-assisted analysis with
appropriate guardrails.

## Keyword Search

Keyword search uses the local search index to find entities by text content.

```powershell
# Search for entities mentioning a specific term
python scripts\search.py "authority ambiguity"

# Search for entities related to a program
python scripts\search.py "program-name"
```

Use keyword search to:
1. Find entities that mention a term but do not wikilink to the corresponding entity file.
2. Discover potential relationships based on shared vocabulary.
3. Locate all entities in a domain for synthesis proposals.

Limitations:
- Keyword search is literal; it does not understand synonyms or related concepts.
- Results are ranked by text match, not semantic similarity.
- Use multiple search queries with different terms to improve coverage.

## Vector Search

Vector search uses embedding-based similarity to find semantically related entities.

```powershell
# Validate the vector search index
python scripts\validate-vector-search.py
```

Vector search is useful for:
1. Finding entities that discuss similar topics using different vocabulary.
2. Discovering relationships that keyword search would miss.
3. Grouping entities by semantic similarity for community detection.

Limitations:
- Vector search requires a pre-built index. Run `python scripts\build-vector-index.py`
  before using vector search if the index does not exist.
- Vector similarity is not the same as factual relatedness. Results should be reviewed
  by the user before creating new relationships.
- Vector search results include a similarity score; use a threshold appropriate for the
  use case (typically 0.7 or higher for strong matches).

## Related but Unlinked Nodes

A key maintainer workflow is finding entities that are semantically related but lack
wikilinks between them. This indicates a gap in the graph's relationship coverage.

Detection process:
1. For each entity, run keyword and vector searches using the entity's key terms.
2. For each search result, check whether a wikilink already exists between the entities.
3. If no wikilink exists but the entities are clearly related, add to the synthesis
   proposal list.

Example:
```powershell
# Find entities related to "command and control"
python scripts\search.py "command and control"
# For each result, check if [[entity-result]] appears in the source entity's file
```

## Community Detection Heuristics

Community detection identifies clusters of entities that are densely interconnected but
sparsely connected to the rest of the graph. These clusters often represent natural
domains or topic areas.

Heuristic approach (no graph visualization tools required):

1. **Wikilink density analysis**: For each entity, count the number of shared wikilink
   targets with other entities. Entities with many shared targets likely belong to the
   same community.

2. **Domain keyword clustering**: Group entities by shared keywords in their titles and
   body text. Entities that share 3+ significant keywords likely belong to the same domain.

3. **Source overlap**: Entities that cite the same source documents are likely related.

4. **Entity type patterns**: Communities often follow patterns like "org + programs it
   runs + people who lead them + decisions that affect them."

For each detected community:
- Assess whether a domain summary page would be useful (see `active-synthesis.md`).
- Check for missing relationships within the community.
- Identify the community's boundary (which entities are on the edge and might connect
  to other communities).

## LLM-Assisted Query Guardrails

LLM tools (including the maintainer skill itself) can be used to analyze graph content,
but their output must be treated as **analysis aids, not graph truth**.

Rules:
1. **LLM output is not a source.** Do not add LLM-generated facts to entity files
   without independent source verification.
2. **LLM suggestions require human review.** Proposed relationships, entity merges, or
   content additions generated by LLM analysis must be added to the review queue.
3. **LLM analysis should be documented.** Record what questions were asked, what the
   LLM suggested, and whether the suggestions were verified.
4. **LLM should not modify the graph directly.** The LLM produces proposals; the user
   or a verified source confirms them.

Allowed LLM uses:
- Summarizing entity content for the maintenance audit report.
- Suggesting potential relationships based on entity content analysis.
- Drafting proposed page outlines for synthesis proposals.
- Generating search queries to find related entities.

Prohibited LLM uses:
- Automatically writing or editing entity files.
- Resolving contradictions without source evidence.
- Merging or deleting entities without user confirmation.
- Creating facts that are not supported by source documents.

## Future Infrastructure Notes

The following capabilities are documented as future infrastructure needs. They are **not**
required for the current maintainer skill and should not be implemented as part of this
skill creation track.

### MCP Server

A Model Context Protocol (MCP) server could provide structured access to the knowledge
graph for external tools. Benefits:
- Standardized query interface for the graph.
- Integration with other tools and workflows.
- Programmatic access to graph metrics and search.

Status: Future consideration. Not required for skill creation.

### Graph Visualization

Interactive graph visualization would help users understand entity relationships and
identify clusters visually. Benefits:
- Visual identification of orphan nodes and disconnected subgraphs.
- Interactive exploration of entity neighborhoods.
- Export of visual reports for briefings.

Status: Future consideration. Tools like Obsidian's graph view provide basic
visualization. Advanced visualization would require new tooling.

## Error Recovery

- If `search.py` returns no results for a query that should match, check that the index
  is up to date: run `python scripts\build-index.py` and retry.
- If `validate-vector-search.py` reports a corrupted index, rebuild it:
  `python scripts\build-vector-index.py`.
- If vector search is not available (no index built), fall back to keyword search only
  and note the limitation in the audit report.
