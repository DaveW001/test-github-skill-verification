# Markdown PDF Publisher

Generate professional PDFs from Markdown using Node.js + Vivliostyle CLI. Full control over cover pages, running headers/footers, page numbers, fonts, and print typography.

## When to Use This Skill

- "Generate a PDF from this markdown document"
- "Publish this proposal/report/white paper as a PDF"
- "I need a print-ready version of this document"
- "Create a branded PDF with headers and footers"

## Stack

| Tool | Purpose |
|------|---------|
| `markdown-it` | Markdown → HTML conversion |
| `markdown-it-footnote` | Footnote/endnote support |
| `@vivliostyle/cli` | HTML → PDF (paged media engine) |
| `serve` (npm) | Local HTTP server (Vivliostyle requires a URL) |
| Python Playwright | Visual QA screenshots |

## Directory Structure

Create a `publication/` folder alongside the source document:

```
[project]/
├── drafts/
│   └── document.md          ← source
└── publication/
    ├── build.js             ← MD → HTML converter
    ├── package.json
    ├── index.html           ← generated
    ├── fonts/               ← local font files (woff2)
    ├── images/              ← auto-copied from source
    ├── styles/
    │   ├── screen.css       ← browser preview
    │   └── print.css        ← Vivliostyle paged media
    ├── output/
    │   └── document.pdf     ← final output
    └── capture-screenshots.py
```

## Step-by-Step Process

### Step 1: Set Up `package.json`

```json
{
  "name": "publication",
  "scripts": {
    "build:html": "node build.js",
    "serve": "npx -y serve . -l 3099 --no-clipboard",
    "build:pdf": "npx -y @vivliostyle/cli build http://localhost:3099/index.html --output output/document.pdf --style styles/print.css --timeout 60000"
  },
  "dependencies": {
    "markdown-it": "^14.0.0",
    "markdown-it-footnote": "^4.0.0"
  }
}
```

### Step 2: Create `build.js`

Key responsibilities:
1. Read the Markdown source
2. Fix relative image paths (`../images/` → `images/`)
3. Convert to HTML with `markdown-it` + footnote plugin
4. **Strip cover/metadata content** — find `<h2>Introduction</h2>` and start there
5. Remove duplicate headings generated by the footnote renderer
6. **Merge inline label paragraphs** — bold-only `<p><strong>Label:</strong></p>` blocks followed by a `<p>` get joined onto one line
7. Copy images and brand logo into `publication/images/`
8. Wrap in full HTML template with Vivliostyle running elements

**Critical post-processing regex** (merge bold labels inline):
```javascript
mainContent = mainContent.replace(
  /<p>(<strong>[^<]+<\/strong>)\s*<\/p>\s*\n?<p>/g,
  '<p>$1 '
);
```

**Content starting point** (skip cover block):
```javascript
const introIdx = mainContent.indexOf('<h2>Introduction</h2>');
if (introIdx !== -1) mainContent = mainContent.substring(introIdx);
```

**HTML template structure**:
```html
<body>
  <!-- Running elements (pulled out of flow by Vivliostyle) -->
  <header id="cover-header">...</header>
  <header id="small-header"><img src="images/logo.png" /></header>
  <footer id="print-footer"></footer>

  <!-- Cover page (full-bleed, page-named for @page coverPage rule) -->
  <section class="cover-page">...</section>

  <!-- Content -->
  <article>
    [converted markdown HTML]
  </article>
</body>
```

### Step 3: Write `styles/print.css`

#### Font Setup
Use local `@font-face` declarations — do NOT rely on Google Fonts (fails in offline/Vivliostyle context):

```css
@font-face { font-family: 'Radikal'; src: url('../fonts/Radikal.woff2') format('woff2'); font-weight: 400; }
@font-face { font-family: 'Radikal'; src: url('../fonts/Radikal-Bold.woff2') format('woff2'); font-weight: 700; }
/* etc. */
```

Packaged Agile fonts are at: `C:\development\2025-pa-website\public\fonts\`

#### Page Margins
```css
@page {
  size: 8.5in 11in;
  margin: 0.65in 1in 0.85in;   /* 1" left/right is standard */
  @top-center    { content: element(smallHeader); }
  @bottom-center { content: element(docFooter); }
}
@page:first {
  margin-top: 0;
  @top-center { content: none; }
}
```

#### Header/Footer Bleed — Critical Gotcha
**⚠️ `calc()` does NOT work for bleed inside Vivliostyle margin boxes.** The `@top-center` margin box clips content to the content-width area — `width: calc(100% + 2in)` looks correct in a browser but gets clipped in the generated PDF.

**The only reliable approach** is to use the explicit page width (8.5in for letter) and a negative left margin equal to the left page margin:
```css
#small-header {
  width: 8.5in;       /* explicit page width — NOT calc() */
  margin-left: -1in;  /* left page margin, negated */
  padding: 0 1in;     /* restore internal padding to match margins */
  overflow: visible;  /* prevent margin-box clipping */
}
```
If you change the page size or left margin, update `width` to match the full page width and `margin-left`/`padding` to match the left margin.

#### Font Size — Critical Gotcha
Do **NOT** set `font-size` on `html` or `body` at all. Vivliostyle inherits and applies scaling differently from a browser, and any `pt` value on the root element may render significantly larger than expected in the PDF.

Set `font-size` **only on `article`** with an explicit `pt` value. Calibrate against your table font size (which is easier to judge) — if tables at `9pt` look right, body text at `9.5pt` will be proportional.
```css
html, body { /* no font-size — leave unset */ }
article    { font-size: 9.5pt; }   /* calibrate to your table size */
article table { font-size: 9pt; }  /* anchor size — what looks right in PDF */
```

#### Cover Page
Use a named page with zero margins and suppress running elements:
```css
.cover-page { page: coverPage; }

@page coverPage {
  margin: 0;
  @top-center    { content: none; }
  @bottom-center { content: none; }
}
```
Extend cover to full bleed using negative margins equal to the page margins:
```css
.cover-page {
  width: calc(100% + 2in);
  margin-left: -1in;
  margin-top: -0.65in;
}
```

#### Endnotes — Inline [n] with Text
The `markdown-it-footnote` plugin wraps each footnote in `<li class="fn-item">`. The `[n]` number is a `<span class="fn-num">` and the text content is a `<p>`. Make them inline:
```css
.fn-item { display: block; margin: 0 0 0.3rem 0; }
.fn-item p { display: inline; margin: 0; padding: 0; }
```

#### Line Height for Print
Browser default line-height (`1.5–1.6`) is too loose for print documents. Use `1.35` for body text. This alone has a significant visual impact on perceived document quality and density.
```css
article { line-height: 1.35; }
```

#### Appendix Page Breaks
CSS `break-before: page` on a heading only works if the heading carries a distinguishing class. Since Markdown doesn't support classes natively, inject the class in `build.js`:
```javascript
// In build.js post-processing:
mainContent = mainContent.replace(
  /<h2>(Appendix[^<]*)<\/h2>/g,
  '<h2 class="appendix-start">$1</h2>'
);
```
```css
/* In print.css: */
article h2.appendix-start { break-before: page; }
```

### Step 4: Build and Serve

```bash
# Install dependencies (first time only)
npm install

# Build HTML
node build.js

# Serve locally (keep running in background)
npx -y serve . -l 3099 --no-clipboard

# Generate PDF (run in a second terminal)
npx -y @vivliostyle/cli build http://localhost:3099/index.html \
  --output output/document.pdf \
  --style styles/print.css \
  --timeout 60000
```

> **Windows Background Server**: Use `cmd /c start /b npx -y serve . -l 3099 > serve.log 2>&1` if you need it non-blocking.

> **⚠️ CRITICAL:** Never run more than one Vivliostyle build at a time. Concurrent builds corrupt the output PDF and can crash Windows Explorer. If the output file is locked (EBUSY error), write to a new versioned filename instead of killing processes. See "Reliable Build & Serve Commands (Windows)" below for the full proven sequence.

### Step 5: Visual QA of the Actual PDF

Playwright screenshots of the local HTML server are not accurate representations of the final PDF (margins, line-heights, and font scaling behave differently under Vivliostyle's print rendering engine). 

**Instead, generate the PDF first and visually validate the actual output using PyMuPDF (`fitz`):**

```bash
pip install pymupdf
```

Create a simple `pdf_to_img.py` script:
```python
import fitz
import sys

pdf_path = sys.argv[1]
page_num = int(sys.argv[2])
out_path = sys.argv[3]

doc = fitz.open(pdf_path)
page = doc.load_page(page_num)  # 0-indexed integer
pix = page.get_pixmap(dpi=150)
pix.save(out_path)
print(f"Saved {out_path}")
```

Extract pages for visual review:
```bash
# Extract Page 2 to an image (page index 1)
python pdf_to_img.py output/document.pdf 1 page2_verify.png
```
You can then look at the .png files using your multimodal models or user-verification to confirm that font sizes, line heights, and margins look perfect before handing it back to the user.

## Mistakes Made — Lessons Learned

### 1. ❌ Relying on `font-size` on `body` for print sizing
**Problem:** Setting `font-size: 12pt` on `html`/`body` was rendering at ~18pt in the Vivliostyle PDF output due to viewport-to-print scaling differences.  
**Fix:** Set `html, body { font-size: 10pt }` as a root anchor, then override explicitly on `article { font-size: 12pt }`.

### 2. ❌ Header/footer bleed width didn't account for changed margins
**Problem:** Header was missing right-side bleed because `width: calc(100% + 1.5in)` was calculated for 0.75" margins but we later changed to 1" margins.  
**Fix:** The bleed formula is always `width: calc(100% + LEFT + RIGHT)` and `margin-left: -LEFT`. When you change `@page` margins, always update the header and footer bleed values to match.

### 3. ❌ Google Fonts link in HTML instead of local fonts
**Problem:** The HTML template included a Google Fonts `<link>` for Inter, which was ignored when Vivliostyle built the PDF. The brand font (Radikal) was declared only in `screen.css` via `@font-face` using relative paths, but the PDF build uses `print.css` as the override stylesheet.  
**Fix:** Embed `@font-face` declarations directly in `print.css` with correct relative paths. Remove the Google Fonts link. Copy the font `.woff2` files into `publication/fonts/`.

### 4. ❌ Split logic was too shallow — cover metadata leaked into body
**Problem:** The initial cover/body splitting logic looked for the first `<hr>` tag, but the Markdown had multiple `<hr>` separators and the content after the cover title (Prepared by, Date) leaked into the article body.  
**Fix:** Anchor the split on a unique heading string: `mainContent.indexOf('<h2>Introduction</h2>')`. Everything before that index is discarded.

### 5. ❌ Duplicate "Endnotes" heading
**Problem:** The Markdown source had `## Endnotes` as a section heading, and the `markdown-it-footnote` renderer also inserted its own `<h3>Endnotes</h3>`. Result: two consecutive Endnotes headings in the PDF.  
**Fix:** Strip the markdown-generated one with: `mainContent.replace(/<h2>Endnotes<\/h2>\s*(<hr>)?/gi, '')`.

### 6. ❌ Bold-label paragraphs rendered as their own block line
**Problem:** Labels like `**A Common Pattern We See:**` in the Markdown are separated from the following paragraph by a blank line, so `markdown-it` renders them as standalone `<p>` blocks. In print, this wastes a full line.  
**Fix:** Post-process the HTML to merge bold-only paragraphs into their following paragraph:
```javascript
mainContent = mainContent.replace(
  /<p>(<strong>[^<]+<\/strong>)\s*<\/p>\s*\n?<p>/g,
  '<p>$1 '
);
```

### 7. ❌ Playwright browser subagent failed due to missing `$HOME`
**Problem:** The browser subagent tool (which uses Playwright internally) failed with a "browser has been closed" error. The cause was a missing `$HOME` environment variable on Windows.  
**Fix:** Always set `$env:HOME = $env:USERPROFILE` before running Playwright scripts on Windows. Add this to the top of any Python Playwright script too: `os.environ.setdefault('HOME', os.environ.get('USERPROFILE', ''))`.

### 8. ❌ Running elements used wrong named-page approach for cover
**Problem:** The initial print.css had `@page :first` (with a space before the colon) instead of `@page:first`. Vivliostyle didn't suppress the header on the cover page, causing the running header to appear on top of the cover gradient.  
**Fix:** Use `@page:first` (no space). Better still, use a named page (`page: coverPage`) with a separate `@page coverPage { @top-center: none }` rule — this is more reliable and decouples cover styling from first-page detection.

### 9. ❌ `calc()` bleed in Vivliostyle margin boxes is clipped
**Problem:** `width: calc(100% + 2in); margin-left: -1in` looks correct in a browser preview but the Vivliostyle margin box clips the content to the content-width area. The result is a header/footer that is left-aligned or missing the right-side bleed in the actual PDF, even though it looks fine in screenshots.  
**Fix:** Instead of trying to use negative margins to bleed out of the box, map the header/footer perfectly to the text width (e.g. `width: 6.5in; margin-left: 0; padding: 0`).

### 10. ❌ Font size still too large even after moving it off `body`
**Problem:** Setting `html, body { font-size: 10pt }` as a root anchor and `article { font-size: 12pt }` still rendered much larger than expected in Vivliostyle because Vivliostyle parses `screen.css` and `print.css` cascaded default elements.
**Fix:** Reset the root explicitly: `html, body { font-size: initial !important; }`. Then set `font-size` **only on `article`** (e.g., `font-size: 10pt`), and critically, set `line-height` tighter for print (e.g. `1.15`).

### 11. ❌ Figure Captions Break Across Pages and Overlap
**Problem:** A standard `<p><img></p><p><em>Caption</em></p>` sequence can break across a page, leaving the image on one page and the caption on the next. Furthermore, negative margins on caption text will cause it to overlap the image.
**Fix:** In `build.js`, write a regex capable of processing multi-line structures (using `[\s\S]*?`) to wrap the image and caption into a `<figure>` and `<figcaption>`. Apply positive margins to the figcaption to push it away from the image, and use `break-inside: avoid;` on the figure.

### 12. ❌ Line height too loose for print
**Problem:** `line-height: 1.55` creates excessive whitespace in a print document, making it look double-spaced.  
**Fix:** Use `line-height: 1.15` and paragraph spacing of `11pt` for print body text. Apply it explicitly to `article`, `article p`, and `article li`.

### 13. ❌ Appendix page breaks required class injection in `build.js`
**Problem:** CSS `break-before: page` on `article h2` matched all H2 headings, not just appendix ones. There is no native Markdown syntax for adding CSS classes to headings.  
**Fix:** Post-process the HTML in `build.js` to inject an `appendix-start` class on H2s whose text starts with "Appendix". Target that class in `print.css`.

### 14. ❌ Running elements (like footers) ignore CSS counter-resets if outside the reset scope
**Problem:** Attempting to `counter-reset: page 1` on the `<article>` tag failed to reset the page numbers in the footer because `<footer id="print-footer">` was placed globally outside the `<article>` tag. In Vivliostyle, running elements inherit the counter scope of their exact DOM position before they are pulled into the margin boxes.
**Fix:** Move the `<footer id="print-footer">` physically *inside* the `<article>` tag in `index.html` (via `build.js`). This ensures it inherits the scoped counter reset, allowing Introduction to be Page 1.

### 15. ❌ Cover page full bleed leaves white margins
**Problem:** Vivliostyle applies default page margins, so setting a background on `.cover-page` leaves white borders on the edges of the PDF.
**Fix:** In `print.css`, apply `@page coverPage { margin: 0; }` to zero out the page box margins completely. Then, make sure the `.cover-page` element covers the full width and height with `width: 8.5in; height: 100vh; margin: 0; box-sizing: border-box;`. Do NOT use negative margins or `calc()` hacks for bleeds.

### 16. ❌ Adjacent H2 and H3 elements have excessive spacing (e.g. "Appendix B")
**Problem:** Because `build.js` strips markdown `<hr>` rules globally to solve one bug, adjacent headings collapse their bottom and top margins (e.g., `1rem` bottom + `1.5rem` top = `1.5rem` gap), creating too much whitespace between "Appendix B" and "Company Overview".
**Fix:** Add an explicit CSS sibling selector in the stylesheet to tighten the gap: `article h2 + h3 { margin-top: 0.5rem; }`.

### 17. ❌ Footer sits too high on the page or gray line floats away from copyright text
**Problem:** Setting large bottom margins on `@page documentBody` (like `margin-bottom: 1.3in`) pushes the entire footer block up, creating an awkward gap at the bottom of the page. Meanwhile, large `padding-top` values inside the `@bottom-left` margin box push the gray top border away from the text inside it. 
**Fix:** To keep the footer low on the page and the text snuggled tightly under the gray line, set a narrow page margin (e.g., `margin-bottom: 0.5in`) combined with a tight padding on the footer block itself:
```css
@page documentBody {
    margin-bottom: 0.5in;  /* Keeps the footer down at the bottom of the page */
    @bottom-left {
        padding-top: 0.08in; /* Keeps text tightly coupled to the top border */
    }
}
```

### 18. ❌ Greedy figure-caption regex consumed entire sections
**Problem:** The `build.js` regex for merging `<figure>` elements with their `<em>Figure N</em>` captions used `[\s\S]*?` between the `<figure>` open/close tags. Even with the lazy `*?` quantifier, this allowed the regex to span across multiple paragraphs and H3 headings, swallowing entire Differentiator sections (e.g., Differentiator 4 disappeared from the rendered HTML).
**Fix:** Replace `[\s\S]*?` with `\s*` so the regex only matches whitespace between the tags, not arbitrary content:
```javascript
// ❌ BAD — greedy, eats across paragraphs:
/<p>\s*<figure class="doc-figure">[\s\S]*?<img src="([^"]+)"[\s\S]*?<\/figure>\s*<\/p>/g

// ✅ GOOD — tight, only matches whitespace:
/<p>\s*<figure class="doc-figure">\s*<img src="([^"]+)" alt="([^"]*)" \/>\s*<\/figure>\s*<\/p>/g
```
**Rule:** Never use `[\s\S]*?` in post-processing regexes unless you are certain the matched structure cannot span across unrelated content blocks.

### 19. ❌ `markdown-it-footnote` ignores source placement of footnote definitions
**Problem:** Moving `[^1]:` definitions to a different location in the Markdown source has zero effect on the rendered output. The plugin always collects all footnote definitions and renders them in a single `<section class="footnotes">` block appended to the very end of the HTML output.
**Fix:** Post-process the rendered HTML in `build.js` to relocate the footnotes section:
```javascript
// Extract the footnotes section and re-insert before the first appendix
const fnMatch = mainContent.match(/<section class="footnotes">[\s\S]*?<\/section>/);
if (fnMatch) {
  mainContent = mainContent.replace(fnMatch[0], '');
  mainContent = mainContent.replace(
    /(<h2 class="appendix-start">)/,
    fnMatch[0] + '\n\n$1'
  );
}
```
Also remove `break-before: page` from `.footnotes` in `print.css` if you want endnotes to flow inline on the same page as the preceding content.

### 20. ❌ Suppressing the footnote back-reference arrow (`↩︎`)
**Problem:** The `markdown-it-footnote` plugin renders a `↩︎` back-reference link at the end of each footnote. In a printed PDF, these local anchor links are meaningless and clutter the endnotes section.
**Fix:** Override the `footnote_anchor` renderer rule in `build.js`:
```javascript
md.renderer.rules.footnote_anchor = () => '';
```

### 21. ❌ EBUSY file lock prevents PDF overwrite
**Problem:** `EBUSY: resource busy or locked` error when Vivliostyle tries to write the output PDF. This happens when the previous PDF is open in a viewer (Acrobat, Edge, Explorer preview) or when an old Vivliostyle process still holds the file handle.
**Fix:** Write to a new versioned filename (e.g., `transcom-response-v10.pdf`) instead of overwriting the locked file. **Do NOT kill Chrome, Edge, or other user-facing processes** to release the lock — the user may be actively using them. Versioned filenames are harmless and avoid data loss.

### 22. ❌ Multiple concurrent Vivliostyle builds crash Windows Explorer
**Problem:** Running multiple `vivliostyle build` commands simultaneously (e.g., from retried or forgotten background processes) corrupts the output PDF. When Windows Explorer opens the folder containing the corrupted PDF, its Preview Handler tries to generate a thumbnail, triggers an Access Violation in `KERNELBASE.dll`, and crashes `explorer.exe` in a continuous loop.
**Fix:**
1. **Never run more than one Vivliostyle build at a time.** Always check for existing build processes before starting a new one.
2. If Explorer starts crash-looping, rename the corrupted `.pdf` files to `.pdf.bak` so the preview handler skips them, then restart Explorer.
3. Use `Get-Process -Name node -ErrorAction SilentlyContinue` to check for lingering processes rather than blindly killing them.

### 23. ❌ Process management — do NOT kill user processes
**Problem:** Using `Stop-Process -Name chrome, msedge -Force` or `taskkill /F /IM chrome.exe` to resolve port conflicts or file locks kills the user's active browser sessions, losing tabs and unsaved work.
**Fix:**
1. **Never kill `chrome.exe` or `msedge.exe`** unless explicitly asked by the user.
2. If a port is in use, start the server on a different port (e.g., `3101`, `3102`, `3103`).
3. If a PDF file is locked, write to a new versioned filename.
4. Only kill `node.exe` processes if you are certain they are orphaned Vivliostyle/serve processes, and even then prefer a clean approach like stopping specific background command IDs.

## Reliable Build & Serve Commands (Windows)

The proven, reliable command sequence for Windows (non-interactive terminal):

```powershell
# 1. Build HTML from markdown
npm run build:html

# 2. Start the local server in the background (use a fresh port if needed)
cmd /c start /b npx -y serve . -l 3099 --no-clipboard > serve.log 2>&1

# 3. Wait for server to be ready
Start-Sleep -Seconds 5

# 4. Generate PDF (use a UNIQUE filename to avoid EBUSY locks)
npx -y @vivliostyle/cli build http://localhost:3099/index.html `
  --output output/document-v1.pdf `
  --style styles/print.css `
  --timeout 60000
```

**Key rules:**
- Do **NOT** chain commands with `&&` or `;` in PowerShell — the CLI security parser rejects them as unsafe. Run each step sequentially.
- Use `cmd /c start /b` for background scripts (like the local server) — `Start-Process` fails for CLI tools that are scripts.
- Always use a **unique output filename** (append version number) to avoid file-lock collisions.
- If the server port is busy, increment to `3101`, `3102`, etc. — do not kill processes to free the port.

### 24. ❌ Wrapped figures not anchored to intended page
**Problem:** A wrapped figure (image + caption) appears one page later than intended, or the heading below slides beside the image.  
**Fix:**
1. Place the `<figure>` **before** the paragraph/list where wrapping should begin.
2. Keep image and caption in the same `<figure>` element.
3. Use `break-inside: avoid; page-break-inside: avoid;` on the figure.
4. Add a clear element before the next heading:
```html
<div class="fig-wrap-clear"></div>
```
```css
div.fig-wrap-clear { clear: both; }
```
5. If still pushed to next page, reduce width by 10-20% or move the figure anchor a few paragraphs earlier.

### 25. ❌ Variant print styles broke cover/header/footer behavior
**Problem:** Layering ad-hoc secondary stylesheets changed `@page` behavior and removed expected running header/footer or altered cover bleed.
**Fix:** For stable production builds, use a **single consolidated print stylesheet** per variant (e.g., `print-wrap-variant.css`) that preserves the baseline `@page` model and only adds targeted figure-wrap classes.
