### Deviation: VACUUM INTO instead of the literal `backup()` API

The plan specified the node:sqlite `backup(new DatabaseSync(src), dst)` API. On Node v24.12.0, that top-level `backup(sourceDb, path)` function does NOT materialize a clean destination file directly: the destination file stays 0 bytes on disk while the copied pages reside in an in-memory/attached representation readable only via the same connection (empirically verified on a synthetic DB: dst file size 0 yet a fresh `SELECT COUNT(*)` returned the rows; a subsequent `wal_checkpoint(TRUNCATE)` + `journal_mode=DELETE` + close was required to flush the data into the main file). For a 14 GB DB that checkpoint-dance adds I/O and leaves an ambiguous intermediate state.

Substitution: `VACUUM INTO '<dst>'` executed on a **read-only** source connection. This is a native SQLite snapshot operation (NOT a `Copy-Item` of a live WAL DB), produces a single transactionally-consistent clean file, needs no WAL reconciliation, and satisfies the plan's Risk Control #1 intent ("non-destructive SQLite snapshot, NOT a Copy-Item of the live DB"). The destination was independently re-verified (quick_check ok + identical row counts), so the safety guarantee the acceptance check depends on is preserved. The earlier empty `...20260714212657.db` artifact from the first read-only `backup()` attempt was removed (it was 0 bytes).
