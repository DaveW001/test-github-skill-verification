## Risks and controls

1. **Shared-database data loss** - P0.2 takes a non-destructive SQLite `backup()` snapshot (NOT a `Copy-Item` of the live DB) BEFORE the upgrade; `quick_check` must return `ok` or the track stops. No schema surgery or storage reset in the primary path.
2. **Multiple installs reintroduce skew** - P1.1 inventories EVERY `opencode*` binary and explicitly stops the Desktop variant (which holds the DB) before the CLI upgrade. The post-upgrade version check is run in a FRESH `pwsh` to defeat cached PATH resolution.
3. **Provider failure misdiagnosed as session failure** - P2.1 retries against an enumerated `minimax*` identifier; if ALL MiniMax identifiers fail with provider errors (not `seq`), P2.1 falls back to a non-MiniMax identifier to prove session persistence, then separately diagnoses MiniMax.
4. **Headless-server workaround misrepresented** - P2.2 probes `/global/health` and `/doc` explicitly and asserts a `POST /session` line in the server log before claiming the attached path works. P2.3's decision criterion forces an explicit adopted / not-adopted verdict and forbids claiming `serve` repaired a database defect.
