## Quality concerns surfaced (not Blocking)

1. **The humanizer source is at a temp-checkout path, not a stable location.** If the temp dir is cleaned up before Phase 3 runs, all of Phase 3 fails. Mitigation: Phase 0.4 precondition gate; if the path is missing, mark Phase 3 DEFERRED. The reviewer recommends the executor also document a "where to recover the humanizer" note in the execution log.
2. **The brand-voice.md in humanizer references is brand-coupled to "Packaged Agile" (not generic).** This is a portability finding the audit will surface. The reviewer flagged it as a documented observation, not a blocker. If the team wants a generic humanizer, that is a separate humanizer-rewrite track.
3. **The operations guide section 7 currently ends at line 145 with the test-2 PASS bullet body; the line `Real global config untouched. Sandbox removed.` is the last content of section 7.** The renumbering algorithm relies on inserting new content after this line. If the operations guide is modified between review and execution (e.g., Dave adds new tests), the algorithm may need adjustment. Mitigation: the executor should verify the section 7 boundary line before insertion.
4. **The plan's PowerShell `Set-Content -Encoding utf8` writes a UTF-8 BOM by default.** This can affect downstream tools. Mitigation: when writing the metadata.json file, use `Set-Content -Encoding utf8NoBOM` (PowerShell 6+) or `[System.IO.File]::WriteAllText($path, $content, (New-Object System.Text.UTF8Encoding($false)))` to write without BOM. The reviewer-applied Phase 6.2 uses the latter pattern.
5. **The `humanizer-cleanup-test-` temp dir in Phase 3.4 uses `Remove-Item -Recurse -Force` in a `finally` block.** This is correct, but the `try` block does not catch all errors (e.g., `Copy-Item` permission errors). If the cleanup fails, the temp dir may persist. Mitigation: the temp dir uses a random GUID prefix so collisions are impossible; the next run will overwrite.
6. **The `tracks.md` and `tracks-ledger.md` upsert is a single-row operation.** If the row text changes (e.g., the title is updated), the `[string]::Replace()` literal substitution may fail. Mitigation: the executor should snapshot the files before the edit and verify the result with `git diff --no-index` per the artifact-output-format decision tree.
7. **The new Phase 6.2 metadata.json update uses `task_count` set to the count of `- [x]` and `- [ ]` boxes, but the plan also has `- [~]` (in-progress) boxes (e.g., the documentation footer).** The reviewer-applied check uses `(?m)^\s*-\s+\[[ x~]\]\s+` which counts all three states. This is correct - the in-progress box in the documentation footer should count. However, the plan's own documentation says "Checkbox states: - [ ] Pending, - [~] In Progress, - [x] Completed" which the executor may misinterpret. Mitigation: the Phase 6.1 check explicitly counts all three states.
