### Phase 0

#### Task 0.1 - Create the redacted inventory utility at `inventory-session-db.ps1` - **Blocking**

**Summary:** The script's purpose and parameter list are correct. Two structural defects in the acceptance check make it unfit for a high-risk privacy track.

**Issues:**
- **B1 (SECURITY).** The required-substring list is `("event.data","candidate-manifest.json","CutoffDays","KeepListPath","SHA256","family")`. The check is `if(($required | Where-Object {-not $t.Contains($_)}).Count){exit 1}` - it requires the script to CONTAIN the literal string `event.data`. The spec says the script must NEVER select, stringify, print, or log `event.data`. The only way to pass the check while honoring the spec is to keep `event.data` in a deny-list constant - but a deny-list reference to `event.data` is the same literal the check requires, so the check is satisfied by the deny-list. The check does not, however, prevent a buggy or malicious script from writing the actual content of `event.data` to a log via `ConvertTo-Json $row.data` or `Write-Output $row.data`, because those source lines do not contain the literal column name `event.data` (they reference `$row.data` or a property named `data`). This is the exact hazard the user named. Replace the static check with a real run + content scan.
- **B2.** The forbidden-substring list is `("SELECT *","Write-Output $row","ConvertTo-Json $row")`. A script that legitimately contains a comment like `# We never use SELECT * on session table` would fail the check. A comment about `ConvertTo-Json $obj` (different variable) would also fail. Use a line-anchored regex (`^\s*SELECT\s+\*\s+FROM` and similar) or AST inspection.
- **B3 (DOC).** The plan says the script "must enumerate table/column metadata first and stop if required mappings are ambiguous". The acceptance check does not verify this behavior.
- **B4 (DOC).** The plan does not say HOW aggregate byte counts are computed without exposing content. SQLite's `length()` (or `octet_length()`) on a BLOB column returns the byte count without exposing the data. Without this specification, the script could read the column content to compute size - which would leak content to the script's process memory (and possibly to logs). The plan must mandate `length()`-style size attribution.
- **B5 (DOC).** The `keep-session-ids.txt` format is not specified. One opaque session ID per line? With comments? Whitespace handling? The acceptance check never exercises the keep-list, so a malformed keep-list could silently include or exclude the wrong sessions.
- **B6 (DOC).** The output JSON schemas (`baseline.json`, `inventory.json`, `candidate-manifest.json`) are referenced by later tasks (0.2, 1.2, 2.2, 3.1) but never formally specified in this task. Downstream acceptance checks fail with cryptic "field missing" errors when the inventory script uses different field names.
- **B7 (RIGOR).** The acceptance check is purely static source-code inspection. The anti-laziness mandate requires active deterministic checks. The script must be run, and the OUTPUT must be inspected.
- **B8 (RIGOR).** The plan's enumeration order says "must enumerate table/column metadata first and stop if required mappings are ambiguous". The acceptance check does not verify that the script does this or stops on ambiguity. The actual `inventory.json` should be inspected for an explicit `schemaMappingStatus: "ok"` field (or a "stop" reason).

#### Task 0.3 - Verify free-space capacity for rollback artifacts - **Blocking**

**Issues:**
- **B13 (CORRECTNESS).** The script uses `Get-PSDrive -Name C` to measure free space, but the plan's error recovery explicitly contemplates the user choosing an alternate backup volume. The check must be parameterized. The dry-run on this host shows C: has 219 GB free and the required is 46 GB, so the current check would pass even though the user could move the backup to a volume with insufficient space.

**Rewritten task 0.3 (full replacement):**

- [ ] **0.3 Verify free-space capacity on the backup volume for rollback artifacts.**

## 4. Cross-cutting concerns

### C1 - Verification rigor (anti-laziness mandate)

The static source-substring acceptance checks for tasks 0.1 and 2.1 are the largest gap. Both tasks REJECT active verification in favor of static source inspection, which means:
- 0.1's "forbidden content" check would pass a script that uses `ConvertTo-Json $row.data` (the script source doesn't contain `event.data`; the variable is named `$row`, not `$event.data`).
- 2.1's "no SQL mutation" check would miss `INSERT`/`UPDATE`/`ALTER` (only `DELETE FROM` and `DROP TABLE` are checked).
- Both checks fail on comments that mention the forbidden patterns in a "do not do this" context.

The rewrites in §3 replace the static checks with multi-part checks: structural source-level checks (line-anchored regexes with comment exclusion) PLUS a real `-WhatIf` run PLUS a content-leak scan of the OUTPUT files. This is the active deterministic check the mandate requires.

### C2 - Schema and format specifications

Several JSON artifacts are referenced by later tasks but not formally specified. The rewrites in §3 add formal schemas to:
- `baseline.json` (redactionPolicy, denyListColumns, attributionSql, inventoriedAt, etc.)
- `inventory.json` (per-session fields, family closure, summary)
- `candidate-manifest.json` (cutoffUtc, policyVersion, summary, sessions[], manifestSha256 sidecar)
- `approval.json` (approvedByUser, manifestSha256, cutoffUtc, policyVersion, approvedAt, dbUnchangedConfirmed, notes?)
- `backup-validation.json` (10+ fields, see 1.1 rewrite)
- `post-delete-validation.json` (10+ fields, see 3.1 rewrite)
- `compact-validation.json` (10+ fields, see 3.2 rewrite)
- `swap-report.json` (5+ fields, see 3.3 rewrite)
- `deletion-log.jsonl` (one JSON per line: sessionId, familyId, manifestSha256, startedAt, finishedAt, status, exitCode, errorMessage?)
- `space-savings.json` (pre/post lengths, preDeletionSessionCount, approvedDeletionCount, applicationSmokeExitCode, applicationSessionListCount, savedBytes, savedGiB)
- `handover.md` (rollback steps, paths, hashes, no IDs)

The Stage 1 plan-creator's spec is a strong foundation; Stage 5 should add a single "Output Schema Reference" appendix to the spec or to the plan that pins these fields exactly.

### C3 - TOCTOU between phases

The plan has a backup in 1.1, then an approval in 1.2, then deletion in 2.2. Between these, the DB could change (e.g., the user starts a new OpenCode session during the user-approval step). The plan's 1.2 and 2.2 rewrites add explicit DB-unchanged checks at each gate. This is defense in depth.

### C4 - Process detection

The current check is `Get-Process -Name 'opencode','OpenCode'`. The CLI is a Node process (npm shim); the Desktop is `OpenCode.exe`; the opencode-server (if running) is `opencode.exe`; the opencode-scheduler is `opencode-job-*.exe`. The check should cover all of these. The rewrite in 0.2 adds an exclusive-open check via `node:sqlite` with `mode=ro` (returns SQLITE_BUSY if a writer holds the DB) - this is a stronger guarantee.

### C5 - Node version preflight

`node:sqlite` requires Node 22.5+ (with `--experimental-sqlite`) or Node 24+ (built-in). The plan does not preflight this. A new task 0.0 was added to the rewrite.

### C6 - RED tests / safety harness

metadata.json declares `test_framework: "PowerShell safety harness plus node:sqlite integration fixtures"` and `test_command: "pwsh -NoProfile -File .conductor\tracks\20260717-opencode-session-db-reduction\tests\run-tests.ps1"`. But the plan has no test tasks. Stage 4 (test-writer) must scaffold this. The plan should reference Stage 4 explicitly.

This is a **bookkeeping vs plan mismatch**. Either:
- (a) The plan should explicitly say "Stage 4 will create the test harness at `tests\run-tests.ps1`" and the executor should verify Stage 4 outputs before Stage 5 starts; OR
- (b) The metadata should set `test_framework: "none"` and `test_command: "n/a"` (since the plan has no tests), and Stage 4 is skipped.

I recommend (a) because the user-stated safety requirements (no direct row deletion, no content exposure) are testable as RED tests. The metadata is correct in declaring the test framework; the plan needs a one-line reference to Stage 4.

## 5. Required user decisions

The following items are uncertain and require user input BEFORE Stage 5 can run. I have not applied changes for these; they are presented to the user.

### U1 - Should the plan include an empirical pre-inventory measurement of the event-sourced duplication claim?

The spec establishes that `event.data` (16.61 GB) and `message.updated.1` (12.63 GB) account for the bulk of the DB. The plan accepts this as a fact and designs around it. A pre-inventory measurement step would run: a small per-session byte attribution test on a known-protected session to confirm `length()` works as expected. This adds a sub-task to 0.1 but improves confidence.

**Recommendation:** YES, add as 0.1a (5 minutes, low risk).

### U2 - Should the deletion gate be per-family (require re-approval between families) or one-shot (one approval covers all families)?

The current plan is one-shot: one manifest hash, one approval, all families deleted in sequence. A per-family gate would require the user to approve each family's deletion hash before the CLI runs. This is more conservative but adds friction.

**Recommendation:** one-shot is acceptable for THIS track (one explicit user approval is the user-stated requirement). Per-family gate can be a future enhancement if the user wants.

### U3 - Should the safety harness (RED tests) be created by Stage 4 or deferred?

Metadata declares `test_framework: "PowerShell safety harness plus node:sqlite integration fixtures"` and `test_command: "pwsh -NoProfile -File ...\tests\run-tests.ps1"`. But the plan has no test tasks.

**Recommendation:** YES, Stage 4 should scaffold the test harness. The plan should reference this with a one-line note. If the user wants to skip Stage 4, set `test_framework: "none"` and `test_command: "n/a"` in metadata.

### U4 - Should the smoke test assert the application session count, or just the exit code?

The plan's smoke test is `opencode session list --max-count 1 --format json` and checks `exit == 0`. The rewrite in F.1 adds a session-count assertion. The user may want to keep the smoke test minimal.

**Recommendation:** include the session-count assertion. It catches a class of silent regressions (CLI list view diverges from DB) that exit-code-only smoke would miss.

### U5 - Should the inventory utility be split into two scripts (read-only inventory + manifest generation) or kept as one?

The current plan has one script. Splitting it would allow the inventory to be re-run without regenerating the manifest, and the manifest to be regenerated without re-inventory. This adds flexibility but also complexity.

**Recommendation:** keep as one script. The DB-unchanged gate (added in 1.2 rewrite) provides the safety net for re-inventory.
