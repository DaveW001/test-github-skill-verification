## Cross-Cutting Notes

### Elevation constraint
Ten tasks use `gsudo pwsh -NoLogo -NoProfile -NonInteractive -Command "..."` and cannot be dry-run from a non-elevated shell. The pattern is well-tested (backtick-dollar escape, single-quote path interpolation, and the `[ordered]@{ ... } | ConvertTo-Json -Compress` accumulator work correctly), but the actual `gsudo` invocation, the registry reads, the schtasks re-registration, and the elevated task API calls need the executor to:

1. Confirm gsudo availability (Task 0.2 - now try/catch-hardened).
2. If gsudo triggers a UAC prompt, the executor must be in a session where the UAC can be approved (or an elevated shell must be opened first and the commands re-run there directly).
3. **List of commands that the executor MUST validate against the real elevated environment:** Tasks 0.2, 1.1, 1.2, 1.3, 1.4, 1.5, 3.2, 3.3, 4.1, 4.2.

### Date handling
The plan correctly captures `$date = Get-Date -Format 'yyyy-MM-dd'` once and reuses it (PowerShell edit hazard #4). The execution log path is derived from `$date` everywhere. Good.

### `gsudo` inner-command escape pattern
The pattern `` `$var `` (backtick-dollar) to prevent the outer PowerShell from expanding a variable that the inner pwsh should see literally is correct. Trace verified for Task 1.1, 1.2, 1.3, 1.4, 1.5, 3.2, 3.3. Single-quote path interpolation inside the inner script (e.g., `'$src'`) is correctly handled because single quotes inside the outer double-quoted `-Command "..."` string are literal characters to the outer PowerShell.

### `<paste command JSON output here>` template pattern
This pattern is repeated in every JSON-output acceptance check. It is workable if the executor:
1. Runs the actual command and captures the JSON output to the terminal.
2. Pastes the JSON literally into the `@' ... '@` here-string.
3. Re-runs the acceptance check.

Risks: manual paste error, accidental inclusion of trailing whitespace, JSON line-break handling inside the here-string. **Acceptable as-is** but I note it as a known fragility. A file-based alternative (write JSON to a temp file, then parse the file) would be more robust - this is a significant refactor that I am **NOT** applying (uncertain, would touch every task). Surfaced below.

### Spec gap - Tier-1 approval definition
The spec uses "Tier-1" but does not define it. I added a cross-reference in Task 3.1 to `references\threshold-policy.md#mid-run-authorization` so the executor knows where to find the policy. **No spec-level change** - the spec's prose is fine; only an in-task pointer was added.
