## Quality Concerns (not Blocking)

The following are minor concerns that did not cause any task to be rated Needs Work, but should be on the executor's radar:

1. **`.Contains()` on potentially-null `$text` (Checks 3, 4, 7, 10, 11, 13).** The plan's pattern `$text = Get-Content -Raw -LiteralPath <path>; $text.Contains('...')` throws `You cannot call a method on a null-valued expression` when the file is missing. In normal execution the file is created by the preceding task, so the throw does not fire; but if an executor runs a check out of order or after a failed write, the error message is misleading. Safer pattern: `$text = Get-Content -Raw -LiteralPath <path> -ErrorAction SilentlyContinue; $ok = $text -and $text.Contains('...')`. Did not apply because the plan is functionally correct in the expected order and the fix touches 7+ checks (systemic change - surface to user if desired).

2. **`gh repo view packaged-agile/skillshare-skills` (Phase 3.3) exit code.** This command exits non-zero with a stderr message when the repo does not exist. The plan does not explicitly capture that. Since the task is explicitly non-blocking and the acceptance check is a literal deferral string, this is cosmetic.

3. **SkillShare `target` command syntax uncertainty (Phase 2.4).** The plan admits `target opencode $targetRoot` may not be the correct syntax and provides a `--help` fallback. The actual SkillShare CLI syntax may differ. The plan handles this with a `target --help` recovery path; the executor must record the actual command in the execution log per the plan's instruction. Acceptable.

4. **Markdown fence literal in task body content (Phases 1.1, 2.3, 3.1).** Task body templates use ` ```markdown ` fences inside the plan, which is correct, but a careless executor might include the fence markers in the file itself. The acceptance checks look for substrings INSIDE the fences, so they would pass either way, but a file with stray fences would be malformed. Acceptable, mentioned in the plan's error recovery for 3.1.

5. **Audit check now matches substring of "skillshare".** After tightening the audit regex, the string `skillshare` (containing `skills` as a substring) would cause a generic `skillshare: 'audit' is not a valid command` error to pass. This is acceptable for a non-blocking informational check; the binary presence is already enforced by Phase 2.1, and Phase 2.5's intent is to capture audit output if it runs.

## Uncertain Items Surfaced to User (NOT applied)

The following items are real but either low-impact or systemic; the orchestrator should decide whether to apply before execution. The reviewer did NOT modify the plan for these.

1. **Systemic `.Contains()` on potentially-null `$text`.** Affects checks 3, 4, 7, 10, 11, 13 (Phases 1.1, 1.2, 2.3, 3.1, 3.2, 4.1). Each currently does `$text = Get-Content -Raw -LiteralPath <path>; $text.Contains('...')`. If the executor runs the check before the file exists (out of order, partial failure), the check throws `You cannot call a method on a null-valued expression` instead of returning False. Safer pattern: `$text = Get-Content -Raw -LiteralPath <path> -ErrorAction SilentlyContinue; $ok = $text -and $text.Contains('...')`. The reviewer estimated this would touch 7+ tasks and chose not to apply unilaterally.

2. **`execution-log-YYYY-MM-DD.md` literal in the plan.** Task 4.3 says "Create `execution-log-YYYY-MM-DD.md` using the actual date". A less capable AI agent could read this literally and create a file named `execution-log-YYYY-MM-DD.md` instead of substituting today's date. The plan's acceptance check uses `-Filter 'execution-log-*.md'` to match any date, which is forgiving, but the literal token is still a footgun. Consider replacing `YYYY-MM-DD` with a placeholder like `$(Get-Date -Format 'yyyy-MM-dd')` in a future review pass.

3. **SkillShare `target` command actual syntax.** The plan uses `& $cmd.Source target opencode $targetRoot` as the primary form and provides a `--help` fallback. The reviewer did not install SkillShare during the review (out of scope and would require an actual `irm | iex`), so the actual syntax was not verified against a running binary. The plan's recovery is correct but the executor should record the exact command in the execution log per the plan's instruction.

4. **Phase 1.1 body content uses single-quoted string with `''` apostrophe escape.** This is the CORRECT PowerShell idiom and DOES return True (verified). The reviewer wants to flag this as a known-fragile pattern for any future reviewer who might change it to a double-quoted string (which would return False because `''` does not escape in double-quoted PowerShell strings). Future review edits should preserve the single-quoted form.
