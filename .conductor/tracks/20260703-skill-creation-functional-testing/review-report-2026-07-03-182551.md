## 2. Critical Finding (the .Contains Bug)

The plan's primary verification pattern is:

```powershell
if (-not (Get-Content -Raw -LiteralPath $c[0]).Contains($c[1])) { $failed += ... }
```

When `$c[0]` does not exist on disk, `Get-Content` writes a non-terminating error and returns `$null`. Then `$null.Contains("foo")` throws `InvalidOperation: You cannot call a method on a null-valued expression.`. In PowerShell, that error short-circuits the `if` condition to `$false` (not `$true`), so the `if` block does **not** run and the missing file is **not** added to `$failed`. The subsequent `if ($failed.Count -eq 0)` therefore outputs `ALL_DELIVERABLE_STRINGS_PRESENT` even when every file is missing.

**Reproduced live in this session** against the real state (5 of 7 expected deliverables absent). The fixed pattern guards every read with `Test-Path -LiteralPath` and an explicit `if ($null -eq $content)` check, so missing files are added to `$failed` and the check correctly reports the failure.

The same anti-pattern is present in 11 other acceptance checks (1.2, 1.3, 2.1, 2.2, 3.1, 3.2, 3.3, 4.1 [output check], 5.1, 5.2, 5.3). All were updated.

**Why this matters for the less-capable executor:** the spec text says these checks should output `True` or `False`. The buggy pattern outputs nothing (empty `$result`) when files are missing - the executor cannot tell pass from fail. After the fix, every check deterministically outputs `True` or `False` (or a specific failure reason).
